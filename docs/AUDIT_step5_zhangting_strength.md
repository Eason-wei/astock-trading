## 一、整体架构

`_compute_zhangting_strength()` 计算 **11 个因子**，分 3 组：

| 组 | 因子 | 含义 | 行号 |
|----|------|------|------|
| **A 时间结构** | A1 首次触板时间 | 距 9:30 的分钟数，-1=未触板 | L503-509 |
| | A2 封板总时长 | 触板后处于涨停价的分钟数 | L512-516 |
| | A3 炸板次数 | 触板后离开涨停价的次数 | L519-536 |
| | A4 最长开板时长 | 单次最长离板分钟数 | L539-552 |
| **B 价格形态** | B1 开盘涨幅 | (开盘-昨收)/昨收 × 100 | L559 |
| | B2 日内振幅 | (最高-最低)/昨收 × 100 | L562-564 |
| | B3 涨停后均价偏离 | 均价越近涨停=越强 | L569-584 |
| | B4 分时平滑度 | R²×50 + 回撤修正×50 | L587-616 |
| **C 量能结构** | C1 涨停比 | 涨停价成交量/全天量(%) | L622-628 |
| | C2 涨停量比 | 触板分钟量/触板前均量 | L632-636 |
| | C3 涨停后量能分布 | 触板后量/全天量(%) | L639-640 |

**综合评分加权**：

```
A1:10% A2:8% A3:10% A4:7% | B1:8% B2:8% B3:12% B4:7% | C1:12% C2:10% C3:8%
= A组35% + B组35% + C组30% = 100%
```

## 二、🔴 Bug / 逻辑错误

### 🔴 P0-1：A3 炸板次数永远=0（死因子）

**位置**：L519-536

**问题**：`zhaban_cnt` 在 L519 初始化为 0，但后续循环只更新了 `in_seal` 状态标记，**没有任何一行递增 zhaban_cnt**。

```python
# L519-536 当前代码
zhaban_cnt = 0
in_seal = False
for i in range(first_hit, n):
    if is_on_limit(i):
        if in_seal is False:
            in_seal = True
            if cur_open > 0:
                max_open = max(max_open, cur_open)
                cur_open = 0
    else:
        in_seal = False   # ← 标记开板了，但没有 zhaban_cnt += 1
        cur_open += 1
```

**影响**：zhaban_cnt 永远=0 → A3_score 永远=100（满分）。炸板 10 次和 0 次得分完全一样。

**权重影响**：A3 占 10%，意味着综合分被压缩到 90% 有效区间。

**修复**：

```python
zhaban_cnt = 0
in_seal = False
for i in range(first_hit, n):
    if is_on_limit(i):
        if not in_seal:
            in_seal = True  # 重新封上
    else:
        if in_seal:
            in_seal = False
            zhaban_cnt += 1  # 封板→开板 = 炸板一次
        cur_open += 1
```

### 🔴 P0-2：B3 均价偏离永远=0（死因子）

**位置**：L569-584, L687

**问题**：B3 只统计了**在涨停价上的成交**（L572 有 `if is_on_limit(i)` 判断），因此均价必然等于涨停价，`relative_vwap` 永远=0。

```python
# L571-574
for i in range(first_hit if first_hit >= 0 else n, n):
    if is_on_limit(i):          # ← 只取涨停价上的分钟
        post_limit_amounts.append(...)
        post_limit_volumes.append(...)

# L584
relative_vwap = (limit_up - post_limit_vwap) / limit_up * 100
# 因为所有纳入均价的价格都=limit_up，所以 vwap=limit_up，relative_vwap=0
```

**影响**：B3_score 永远=100（满分）。占权重 12%，综合分进一步被压缩。

**修复**：如果要衡量"触板后资金承接质量"，应统计触板后**所有分钟**（含开板期间）：

```python
for i in range(first_hit if first_hit >= 0 else n, n):
    # 去掉 is_on_limit 判断，统计所有触板后成交
    post_limit_amounts.append(float(amounts[i]))
    post_limit_volumes.append(volumes[i])
```

这样开板期间低价成交会拉低均价，relative_vwap > 0 → 扣分，才有区分度。

### 🔴 P0-3：A2 封板时长分母 210 不合理

**位置**：L660

```python
a2_score = min(total_seal / 210 * 100, 100)
```

**问题**：`total_seal` 是从 `first_hit` 开始统计的，分母固定 210 意味着"触板后最多封 210 分钟=满分"。但：
- 9:31 封板（first_hit=1）→ 最多封 240 分钟，240/210=114%→cap at 100，但 210 分钟后才满分
- 10:00 封板（first_hit=30）→ 最多封 211 分钟，211/210≈100.5%，勉强满分
- 11:00 封板（first_hit=90）→ 最多封 151 分钟，151/210=71.9%，全程封板只能拿 72 分

**影响**：早盘封板的票得分合理（cap at 100），但中后盘封板即使全天不炸也只能拿 72~90 分。A2_score 对封板质量的评估随 first_hit 非线性衰减。

**修复**：分母应该是触板后剩余分钟数：

```python
available = n - first_hit  # 触板后可用的总分钟数
a2_score = min(total_seal / available * 100, 100) if available > 0 else 0
```

## 三、🟡 设计问题

### 🟡 P1-1：C1 评分方向与注释矛盾

**位置**：L624-625（注释）vs L697（评分）

注释说：
> 封板期间成交量越少（占比越低），说明筹码锁定越好

但评分逻辑：
```python
c1_score = min(seal_ratio / 20 * 100, 100)  # seal_ratio 越大分越高
```

**根因**：`seal_on_limit_vol`（L627）= 触板后且在涨停价上的成交量。这笔量大有两种解读：
- 多头：封单吃掉了大量抛压 → 强
- 空头：涨停板成交多 = 抛压重 → 弱

实际 C3（涨停后总量占比）已经覆盖了"触板后抛压"维度（C3 越大=越弱）。C1 和 C3 方向相反、高度相关，属于信息冗余。

**建议**：
- 方案 A：重定义 C1 为"封板时间占比 = 涨停价分钟数/触板后总分钟数"，衡量封板稳定性
- 方案 B：删除 C1，权重并入 C3

---

### 🟡 P1-2：C1 和 C3 高度冗余

**位置**：L627-628 vs L639-640

| 因子 | 公式 | 含义 |
|------|------|------|
| C1 | 涨停价上成交量 / 全天量 | 封板期间抛压 |
| C3 | 触板后总成交量 / 全天量 | 触板后总抛压 |

C1 ≤ C3 恒成立。当不炸板时 C1=C3。两者占权重合计 20%，存在共线性风险。

**建议**：保留 C3（更全面），C1 改为"封板时间占比"（基于时间而非量）以消除冗余。

---

### 🟡 P1-3：B4 对涨停前分时区分度低

**位置**：L587-616

涨停前分时是从低到高走向涨停的。`max_dd`（最大回撤）在这种"单边上涨"序列中通常很小（<5%），导致 `1 - max_dd/20 ≈ 0.95`，B4 ≈ R²×50 + 47.5。

B4 实质上退化为纯 R² 指标，区分度有限。

**建议**：权重从 7% 降到 3-4%，或替换为更有区分度的指标（如"涨停前加速斜率"）。

---

### 🟡 P1-4：分钟数据午间休市段未处理

A股 11:30~13:00 午休，分钟数据可能有 90 分钟空档。

- 如果数据跳过午休（240 条）：时间索引偏移，first_hit 对应的实际时间不准
- 如果数据包含午休（241 条，午休 vol=0）：A2 封板时长会被午休的 90 个"零量"分钟稀释

当前代码假设分钟数据连续 241 条（9:30~15:00），未做午休处理。

**建议**：确认 MySQL 分钟数据格式后，在计算时间指标时排除午休段。

## 五、`_check_minute_pattern()` 与涨停强度因子的不一致

| 维度 | `_check_minute_pattern()` (L381) | `_compute_zhangting_strength()` |
|------|------|------|
| 一字板判断 | `amp < 0.5` | `amp < 0.5`（隐式） |
| 早盘脉冲 | `f30 > 85 AND amp < 3` | 无对应因子 |
| 早盘拉升 | `f30 > 70` | A1 首次触板时间 |
| 尾盘拉升 | `q4 > 1%` | 无对应因子 |
| 尾盘砸盘 | `q4 < -1%` | 无对应因子 |

`_check_minute_pattern()` 的形态分类结果被 `run()` 函数用于综合评分（L323-330 的 FORM_SCORES），但它与 `_compute_zhangting_strength()` 是**完全独立的两套分析体系**，因子不互通、阈值不同步。
